# Willa 组件路线

这个文档只记录尚未实现、尚未公开或接口仍待确认的组件方向。已经作为公开组件进入
示例站的内容不再放在路线清单中；组件创建、CSS、主题变量和示例文档规则以
[组件指南](./component.md)、[CSS 指南](./css.md) 和
[代码风格指南](./style.md) 为准。

Willa 面向 AI 产品、博客、文档站、内容平台、MDX 页面和富交互内容渲染。AI 产品是
当前新增主线，但不是替代博客和文档站目标；后续组件规划需要同时服务产品交互和
富内容表达。

## 规划原则

| 原则       | 说明                                                                                                                                                                           |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 包边界清晰 | 布局底座归 `@willa-ui/layout`；通用展示底座归 `@willa-ui/content`；通用表单底座归 `@willa-ui/form`；AI 场景组件归 `@willa-ui/ai`；外部平台和富媒体场景归 `@willa-ui/widgets`。 |
| 少而完整   | 优先建设能覆盖真实产品常用路径的组件，不为少见场景提前堆参数。                                                                                                                 |
| 可组合     | 高阶组件优先组合现有基础组件，避免把输入、布局、状态和业务流程塞进一个过重组件。                                                                                               |
| 不重复     | 已有基础组件能表达的能力不再新建同义组件；确实需要组合封装时，先明确差异。                                                                                                     |

## 当前结论

现有组件复用方向基本成立：PromptInput 复用 InputPanel，ReasoningSteps 复用
Timeline，Upload 复用 FilePreview 和 Download，说明跨包组合链路可继续沿用。
GenerationCard 和 ToolCallCard 暂不抽公共 API；折叠、状态标记和卡片 chrome
只作为 AI 包内部私有 primitive 的观察方向。

表单选择类已经通过 `useSelectablePanel` 和 `selectablePanelParts` 共享浮层与
面板基础。继续新增 Combobox、AutoComplete 等选择组件前，应优先收敛内部
selection model，覆盖选中值归一化、多选切换、清空和 `renderValue`，避免复制
Picker / TreeSelect 的选择状态逻辑。

CSS 主题归属当前基本合理。ColorPicker 色谱和 Separator 虚线渐变属于结构或功能色，
可以留在组件 CSS；新增主题变量仍必须放在组件所属包的 theme 文件中，不进入 `willa`
聚合包。

已有大 API 组件需要控制继续膨胀。`Tour` 已按文案、交互行为、定位、渲染和样式类名
收敛为分组 API；后续同类组件新增能力时，优先使用稳定配置对象或组合区域，避免继续
平铺根级 props。

## 基础产品组件

基础组件路线只记录还缺的通用产品能力。已经公开的表单、表格、弹层、反馈、文件、
上传下载、代码、引用和内容渲染组件不在这里重复列出。

布局能力优先建设低耦合底座，再考虑可拖拽、可调整尺寸的复杂面板。分栏拖拽类组件
依赖最小宽度、持久化、键盘可访问性、移动端降级和嵌套策略，不能作为布局体系的第一
层抽象。

基础组件缺口参考 Ant Design、MUI 和 Radix Primitives 的通用组件分类，但只保留
符合 Willa 当前定位的高频产品能力。组件落点遵循现有包边界：展示、反馈和导航类放在
`@willa-ui/content`，表单输入类放在 `@willa-ui/form`，布局底座放在
`@willa-ui/layout`。

## 内容与渲染增强

内容组件仍是 Willa 的重要目标。后续增强应优先补齐博客、文档站、MDX 和内容平台中
可复用的能力，而不是只服务单一示例。

| 优先级 | 待补能力   | 说明                                           |
| ------ | ---------- | ---------------------------------------------- |
| P1     | 资源索引   | 面向文档站和知识库的资源列表、分类和搜索组合。 |
| P2     | 富媒体编排 | 多媒体内容的分组、说明、来源和版权信息组织。   |

## 暂缓建设

以下方向需要产品形态更稳定后再设计接口：

| 方向                    | 原因                                                                   |
| ----------------------- | ---------------------------------------------------------------------- |
| `Transfer` 穿梭框       | 更偏企业后台批量选择，当前可由表格、列表和筛选组合覆盖。               |
| 独立 `MarkdownRenderer` | 先评估复用现有 MDX、代码和内容组件，避免重复建设渲染管线。             |
| 完整工作流编辑器        | 容易演化成平台级系统，需要等 Agent 产品形态稳定后再拆分通用组件。      |
| 完整企业治理后台        | 权限、审计、额度和成本管理更偏平台配置，通用 UI 抽象需要真实产品验证。 |
